# Write OKR

为内核/嵌入式开发方向工程师撰写 OKR。

## 触发条件

- 用户使用 `/write-okr` 命令
- 用户要求写 OKR、制定工作目标或规划

## 执行流程

### Step 1: 收集输入

通过 AskUserQuestion 或从用户消息中提取以下信息:

1. **角色**: 如Linux开发工程师、Linux技术负责人
2. **工作方向**: 如 RISC-V arch内核开发、性能优化、驱动开发
3. **已知的重点项目或任务**: 用户正在做或计划做的具体事项
4. **团队/组织目标** (可选): 上级目标用于对齐

如果用户未提供以上信息，查看:
- `.qoder/gen/` 下的调研报告，了解技术关注点
- `patches/` 目录下的补丁，了解开发活动
- 当前仓库跟用户相关的近期 git log，推断工作方向

### Step 2: 生成 OKR

#### OKR 结构

OKR 包含 1 个 Objective，该 Objective 包含 2-6 个 Key Result。

**Objective 撰写原则**:
- 定性描述，有激励性和方向感
- 体现技术价值或业务价值
- 内核开发常见方向: 特性开发、性能优化、质量提升、上游贡献、技术影响力

**Key Result 撰写原则**:
- 一句话描述，量化指标直接嵌入描述中，保持简洁，不需要单独的背景段落
- 有明确的目标和完成标准，有挑战性但可实现

#### 内核开发 KR 度量参考

| 类别 | 可量化指标示例 |
|------|--------------|
| 上游贡献 | 提交/合入 patch 数量、Review patch 数量 |
| 特性开发 | 完成 N 个特性的开发与测试 |
| 性能优化 | 基准测试提升 N%、延迟降低 N% |
| 质量保证 | Bug 修复数量、测试覆盖率、CVE 响应 |
| 文档 | 技术文档/设计文档数量 |
| 协作 | Code review 参与度、技术分享次数 |
| 工具建设 | 自动化工具/脚本数量、CI 流程改进 |

### Step 3: 输出

**终端摘要**: 简洁列出 O 和 KR 的标题

**完整文件**: 保存到 `.qoder/gen/okr/` 目录，文件名格式: `okr-{YYYY}-{主题关键字}.md`

#### 文件模板

```markdown
# OKR-{YYYY}-{主题关键字}

---

撰写日期: {date}

## Objective: {目标标题}

**KR1**: {一句话描述，包含目标、行动和量化指标}
**KR2**: {一句话描述，包含目标、行动和量化指标}
**KR3**: {一句话描述，包含目标、行动和量化指标}

```

注意: 最终文档只包含 O 的一句话标题 + 若干 KR 的一句话描述。

### Step 4: 交互优化

生成初版后，询问用户:
1. 是否需要调整某个 KR 的难度
2. 是否有遗漏的工作方向
3. KR 的量化指标是否合理

根据反馈迭代修改，直到用户满意。每次迭代的修改都增量保存到该文件，用“---”间隔

## 注意事项

- 所有内容使用中文
- KR 必须可量化，便于评价，避免模糊表述如"提升代码质量"
- 至少有一个 KR 应有足够的挑战性
- 如果用户只给了模糊方向，主动从仓库活动中挖掘具体内容来充实 KR
